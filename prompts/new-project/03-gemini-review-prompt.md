## 리뷰 요청 사항

다음 관점에서 검토해주세요:

### 1. 시장/제품 관점
- 타겟 사용자 정의가 충분히 구체적인가?
- 경쟁사 대비 차별점이 실제로 의미 있는가?
- MVP 범위가 적절한가? (너무 크거나 작지 않은가?)
- 놓친 중요한 기능이나 사용자 시나리오가 있는가?

### 2. 기술 아키텍처 관점
- 선택한 아키텍처 패턴이 이 프로젝트 규모에 적합한가?
- 오버엔지니어링 또는 언더엔지니어링된 부분이 있는가?
- 기술 스택 선택에 잠재적 문제나 더 나은 대안이 있는가?
- 스케일링 또는 미래 확장 시 문제가 될 수 있는 설계 결정이 있는가?

### 3. 병렬 개발 관점
- 모듈 간 의존성이 병렬 작업을 방해하는 부분이 있는가?
- 공유 인터페이스 정의가 충분히 명확한가?
- 추가로 분리할 수 있는 영역이 있는가?

### 4. 리스크 관점
- 문서에서 언급되지 않은 기술적 리스크는?
- 서드파티 서비스 의존도가 너무 높은 부분이 있는가?
- 보안 취약점이 발생할 수 있는 설계 결정이 있는가?

### 5. 개선 제안
- 현재 문서에서 수정/보완해야 할 3-5가지 핵심 사항
- 각 제안에 대한 구체적인 대안

## 팁: Gemini를 더 잘 활용하는 방법

1. **두 번 물어보기**: 처음 리뷰 후, "방금 제안한 아키텍처 변경 사항을 더 구체적으로 설명해줘"
2. **대안 요청**: "다른 방식으로 이 문제를 해결한다면 어떤 접근법이 있을까?"
3. **반론하기**: "우리가 이 방식을 선택한 이유는 X이다. 그래도 네 제안이 더 낫다고 생각하나?"
4. **시장 분석 심화**: "이 분야의 최신 트렌드나 성공 사례를 알고 있다면 공유해줘"
