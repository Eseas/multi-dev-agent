# 개발 기획 문서 템플릿

개발 기획 문서는 다음 5개 섹션으로 구성된다.
각 기획 단위(기능/작업)마다 이 형식을 따른다.

> **AI 에이전트 필수 사전 작업**:
> 1. 이 문서를 읽고 코드를 구현하기 **전에**, 해당 서비스의 `tech-stack.md`를 먼저 확인한다.
> 2. 이 기획 문서에서 `tech-stack.md`에 없는 새로운 기술 스펙이 등장하면, **구현 시작 전에 `tech-stack.md`를 먼저 수정**한다.
> 3. **구현 방식은 AI가 자율적으로 결정한다.** 이 문서는 무엇을(What)과 왜(Why)를 정의하며, 어떻게(How)는 명시하지 않는다.

---

## 문서 구조

```
1. 프로젝트 이름
2. 달성 목표
3. 개발 진행 과정 순서    ← API 계약 + 데이터 모델 변경 사항 포함
4. 개별 테스트 내용       ← 테스트 시나리오 목록 (코드 없음)
5. 전체 테스트 내용       ← Playwright E2E 시나리오 흐름 (코드 없음)
```

> 기술 스택 및 환경 정보는 `tech-stack.md`에서 관리한다. (위치: `workspaces/{서비스명}/tech-stack.md`)

---

## 1. 프로젝트 이름

해당 작업이 속하는 서비스/프로젝트명과 기능명을 명시한다.

형식: `{서비스명} - {기능명}`

예시: `Dailyword - 관리자 대시보드 생성`

---

## 2. 달성 목표

이 작업이 완료되었을 때 달성해야 하는 상태를 구체적으로 기술한다.

작성 원칙:
- 측정 가능한 기준 사용
- "~할 수 있다" 형태의 완료 조건
- 기술적 요구사항과 비즈니스 요구사항 분리

---

### 비즈니스 목표
- 관리자가 웹 브라우저에서 사용자 통계를 조회할 수 있다
- 관리자가 콘텐츠를 CRUD 할 수 있다

### 기술적 목표
- API 응답 시간 < 500ms
- 관리자 인증/인가 적용

---

## 3. 개발 진행 과정 순서

목표 달성을 위한 구현 단계를 순서대로 나열한다.
각 단계는 독립적으로 완료 확인이 가능해야 한다.

작성 원칙:
- 단계 간 의존성 명시
- 각 단계의 산출물 명시
- 대상 root project 표기
- 데이터 모델 변경이 있으면 추가/변경 컬럼과 이유를 서술로 명시
- API가 있으면 엔드포인트, 입력, 출력, 에러 케이스를 서술로 명시
- **구현 방법(클래스명, 패턴, 알고리즘 등)은 명시하지 않는다**

---

### Step 1: {단계명}
- **대상**: {root project}
- **작업 내용**:
  - 달성해야 할 것 1
  - 달성해야 할 것 2
- **데이터 모델 변경** (있을 경우):
  - {테이블명}: {추가/변경할 컬럼명}, {타입}, {제약조건}, {이유}
- **API 계약** (API가 있을 경우):
  - 메서드/경로: POST /admin/login
  - 요청: email(string), password(string)
  - 응답: token(string), expiresIn(number)
  - 에러: 401 - 자격 증명 불일치 / 403 - 관리자 계정 아님
- **산출물**: {파일/API/화면 등}
- **의존성**: 없음 / Step N 완료 후

## 4. 개별 테스트 내용

각 Step에 대응하는 테스트 시나리오를 정의한다.
**코드는 작성하지 않는다.** 구현 AI가 시나리오를 보고 자율적으로 테스트 코드를 작성한다.

작성 원칙:
- Step 번호와 매핑
- 사용할 테스트 프레임워크 명시
- [정상] / [이상] / [엣지] 구분
- Given / When / Then 형태로 기술
- 테스트 대상과 기대 결과를 명확히

---

### Step 1 테스트: {테스트 대상}

**프레임워크**: {JUnit / Jest / pytest 등}

| 구분 | Given | When | Then |
|------|-------|------|------|
| [정상] | 유효한 관리자 계정 | 로그인 요청 | 200 + JWT 반환 |
| [이상] | 잘못된 비밀번호 | 로그인 요청 | 401 반환 |
| [이상] | 일반 유저 토큰 | 관리자 API 접근 | 403 + code 7000 반환 |
| [엣지] | 만료된 토큰 | API 접근 | 401 반환 |

---

## 5. 전체 테스트 내용

모든 Step이 통합된 상태에서의 E2E 시나리오를 정의한다.
**Playwright**를 사용하여 구현하되, **코드는 작성하지 않는다.**
구현 AI가 시나리오 흐름을 보고 자율적으로 테스트 코드를 작성한다.

작성 원칙:
- 사용자 관점의 흐름으로 기술
- 사전 조건(필요한 데이터 상태) 명시
- 각 액션과 기대 결과를 순서대로 기술
- Happy Path 우선, 이후 실패 시나리오 추가

---

### E2E 시나리오 1: {시나리오명}

**사전 조건**: {필요한 데이터 상태}

| 순서 | 액션 | 기대 결과 |
|------|------|-----------|
| 1 | 관리자 계정으로 로그인 | 대시보드 페이지로 이동 |
| 2 | 사용자 통계 영역 확인 | 총 회원 수, 신규 가입 수 표시 |
| 3 | 콘텐츠 생성 버튼 클릭 후 제목/내용 입력 | 생성 완료 메시지 표시 |
| 4 | 콘텐츠 목록에서 신규 항목 확인 | 목록에 생성한 콘텐츠 표시 |

### E2E 시나리오 2: {시나리오명} - 실패 케이스

| 순서 | 액션 | 기대 결과 |
|------|------|-----------|
| 1 | 로그아웃 상태에서 대시보드 URL 직접 접근 | 로그인 페이지로 리다이렉트 |
| 2 | 일반 유저 계정으로 로그인 후 대시보드 접근 시도 | 403 에러 페이지 표시 |
