# 개발 기획 문서 템플릿

## 개요

이 문서는 `planning-spec.md` 작성을 위한 템플릿이다.
각 기능/작업 단위마다 이 형식을 따른다.

**에이전트 필수 사전 작업**:
1. 이 문서를 읽고 코드를 구현하기 **전에**, 해당 서비스의 `tech-stack.md`를 먼저 확인한다
2. 이 기획 문서에서 `tech-stack.md`에 없는 새로운 기술이 등장하면, **구현 시작 전에 `tech-stack.md`를 먼저 수정**한다
3. **구현 방식은 에이전트가 자율적으로 결정한다.** 이 문서는 무엇을(What)과 왜(Why)를 정의하며, 어떻게(How)는 명시하지 않는다

---

## 문서 구조

```
1. 기획서 헤더       기능명, N값, 파이프라인 모드
2. 목표              비즈니스 목표 + 기술적 목표 (측정 가능한 기준)
3. 제약사항          기술적/비즈니스 제약, 기존 시스템 통합 포인트
4. 구현 방법 탐색    N개 approach, 각각 핵심 아이디어/장단점/기술스택
5. 개발 진행 순서    단계별 작업 내용, API 계약, 데이터 모델 변경
6. 개별 테스트       단위 테스트 시나리오 (코드 없음)
7. 전체 테스트       E2E 테스트 시나리오 (코드 없음)
8. 대상 Root Project 구현이 반영될 프로젝트
9. 참고사항          관련 이슈, 코드 위치, 외부 문서
```

기술 스택 및 환경 정보는 `tech-stack.md`에서 관리한다. 기획서에는 approach별 특화 스택만 명시한다.

## 작성 시 주의사항

### 반드시 지켜야 할 것

- `탐색할 방법 개수`는 반드시 명시한다 (1, 2, 3 또는 자동)
- 기술 스택은 실제 라이브러리명과 버전으로 명시한다
- 성공 기준은 측정 가능한 형태로 작성한다
- API가 있으면 메서드/경로/요청/응답/에러 케이스를 모두 명시한다
- 데이터 모델 변경이 있으면 어떤 테이블의 어떤 컬럼이 어떻게 바뀌는지 명시한다

### 하지 말아야 할 것

- 클래스명, 메서드명, 특정 디자인 패턴 지정 (에이전트가 결정)
- 파일 구조 지정 (에이전트가 결정)
- 코드 스니펫 작성 (에이전트가 구현)
- "적절한", "빠른", "좋은" 같은 추상적 표현

### 좋은 기획서의 기준

- 이 문서만 보고 에이전트가 구현을 시작할 수 있는가?
- 완료 기준이 객관적으로 확인 가능한가?
- 방법 간의 차이가 명확히 드러나는가?
- 기존 시스템과의 연결 지점이 모두 명시되어 있는가?
