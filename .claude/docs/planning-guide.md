# 기획 가이드 (Phase 0)

## 개요

기획서(`planning-spec.md`)는 파이프라인 전체의 입력이다. 이 문서를 기준으로 Architect가 구현 설계를 생성하고, Implementer가 구현하고, Reviewer/Tester가 평가한다.

**기획서의 원칙**: 무엇을(What)과 왜(Why)를 정의한다. 어떻게(How, 클래스명·패턴·알고리즘 등)는 에이전트가 결정한다.

---

## 기획서 위치

```
workspaces/{서비스명}/planning/in-progress/{기능명}/planning-spec.md   ← 작성 중
workspaces/{서비스명}/planning/completed/{기능명}/planning-spec.md     ← 파이프라인 실행 대기
```

작성이 완료되면 `in-progress/`에서 `completed/`로 이동한다:
```bash
mv workspaces/{서비스명}/planning/in-progress/{기능명} \
   workspaces/{서비스명}/planning/completed/{기능명}
```

## 기획서 전체 템플릿

```markdown
# 기획서: {기능명}

## 1. 목표

### 비즈니스 목표
- 이 기능이 해결하는 문제
- 완료 시 달성해야 할 상태 ("~할 수 있다" 형태)

### 기술적 목표
- 성능 기준 (예: 응답 시간 < 500ms)
- 보안 요구사항
- 기타 측정 가능한 기술 지표

## 2. 제약사항

- 기술적 제약: 기존 프레임워크, 언어 버전, 라이브러리 제한
- 기존 시스템 통합 포인트: 어떤 모듈/API와 연결되는지
- 비즈니스 제약: 하위 호환성, 데이터 마이그레이션 등

## 3. 구현 방법 탐색

**탐색할 방법 개수: {N}개**
<!-- 1, 2, 3 중 하나 지정. 또는 "자동"으로 시스템이 복잡도 기반 판단 -->

**파이프라인 모드: {alternative 또는 concern}**
<!-- alternative: 독립적 대안 비교 (기본값) -->
<!-- concern: 보완적 관심사 통합 -->

### 방법 1: {방법명}
- **핵심 아이디어**: 한 문장 요약
- **예상 장점**: 구체적으로
- **예상 단점**: 솔직하게
- **기술 스택**: 실제 라이브러리명과 버전 (예: Spring Security 6, JJWT 0.12)

### 방법 2: {방법명}
- **핵심 아이디어**:
- **예상 장점**:
- **예상 단점**:
- **기술 스택**:

## 4. 대상 Root Project

이 기획서의 구현이 반영될 Root Project를 명시한다.
예: `{서비스명}-BE`, `{서비스명}-FE`

## 5. 참고사항

- 관련 이슈 번호
- 기존 코드 위치 (예: `src/auth/` 하위)
- 외부 문서 링크
- 의존하는 선행 작업
```

## N값 결정 가이드

### 값별 의미

| N값 | 의미 | 활성화 Phase |
|-----|------|-------------|
| `1` | 단일 구현. 대안 탐색 없음 | Phase 1~3, Simplifier |
| `2` | 두 가지 대안 또는 두 가지 관심사 | Phase 1~4 전체 |
| `3` | 세 가지 대안 또는 세 가지 관심사 | Phase 1~4 전체 |
| `자동` | Architect가 기획서 복잡도 기반으로 1~3 자동 결정 | - |

### 상황별 권장값

| 상황 | 권장 N | 이유 |
|------|--------|------|
| 버그 수정 | 1 | 해법이 명확 |
| 단순 CRUD 기능 추가 | 1~2 | 구현 방향이 거의 정해져 있음 |
| 일반 기능 개발 | 2 | 대안 하나를 비교하는 것만으로 충분 |
| 라이브러리 선택이 중요한 경우 | 2~3 | 옵션 간 trade-off 비교 필요 |
| 아키텍처 결정 | 3 | 중요한 결정, 충분한 탐색 필요 |
| FE + BE 분리 구현 | 2 (concern) | 관심사가 다른 두 구현을 통합 |

## 파이프라인 모드 선택 가이드

### Alternative 모드 (기본값)

N개 approach가 **같은 목표를 다른 방식으로** 구현하는 독립적 대안.
Comparator가 비교하여 가장 좋은 하나를 추천한다.

사용 시나리오:
- "JWT vs Session 방식 중 어느 게 더 나을까?"
- "ORM 직접 쿼리 vs QueryDSL 중 어느 게 더 유지보수하기 좋을까?"

```markdown
**파이프라인 모드: alternative**
### 방법 1: JWT 기반 인증
### 방법 2: Session 기반 인증
```

### Concern 모드

N개 approach가 **서로 다른 관심사를 담당**하는 보완적 구현.
Integrator가 모두 합쳐서 하나의 통합 코드를 만든다.

사용 시나리오:
- approach-1 = 인증 모듈, approach-2 = 비즈니스 로직 모듈
- approach-1 = 백엔드 API, approach-2 = 프론트엔드 화면

```markdown
**파이프라인 모드: concern**
### 방법 1: 인증/권한 처리 (BE)
### 방법 2: 사용자 관리 비즈니스 로직 (BE)
```

## 기획 대화 흐름 (Claude Code와 작업 시)

`planning/in-progress/{기능명}/` 디렉토리에서 Claude Code와 대화하며 기획서를 작성한다.

### 1단계: 요구사항 파악

Claude가 확인할 사항:
- "이 기능이 정확히 무엇을 해결하나요?"
- "완료 시 어떤 상태가 되어야 하나요?"
- "성공을 어떻게 측정할 건가요?"

### 2단계: 제약사항 확인

- "기존 시스템의 어느 부분과 연결되나요?"
- "사용 중인 프레임워크/라이브러리 버전은?"
- "성능이나 보안 요구사항이 있나요?"

### 3단계: 구현 방향 제안

- N개 방법 제안 + 각 방법의 장단점 설명
- N값 및 파이프라인 모드 추천
- 사용자 의견 반영하여 기획서 작성

### 4단계: 기획서 작성 및 완성

- `planning-spec.md` 작성 완료
- 사용자 확인
- `completed/`로 이동

## 자주 하는 실수

### 방법 개수 누락

```markdown
# 나쁜 예
### 방법 1: JWT
### 방법 2: Session

# 좋은 예
**탐색할 방법 개수: 2개**
### 방법 1: JWT
### 방법 2: Session
```

### 구현 방법(How) 지정

```markdown
# 나쁜 예 (에이전트의 자율성을 침해)
- JwtAuthenticationFilter 클래스를 만들어 SecurityFilterChain에 등록

# 좋은 예 (What/Why만 정의)
- JWT 토큰 기반으로 API 접근을 인증/인가할 수 있어야 한다
- 토큰 만료 시 자동으로 갱신되어야 한다
```

### 측정 불가능한 기준

```markdown
# 나쁜 예
- 좋은 성능
- 빠른 응답

# 좋은 예
- API 응답 시간 p95 < 200ms
- 동시 접속자 1,000명 처리 가능
```

### 기술 스택을 추상적으로 명시

```markdown
# 나쁜 예
- **기술 스택**: 적절한 인증 라이브러리

# 좋은 예
- **기술 스택**: Spring Security 6, JJWT 0.12, BCrypt
```

## tech-stack.md와의 관계

서비스의 전체 기술 스택은 `workspaces/{서비스명}/tech-stack.md`에서 관리한다.

기획서에서 `tech-stack.md`에 없는 새로운 기술이 등장하면:
1. 기획서 작성 시 이를 명시한다
2. Implementer가 구현 시작 전에 `tech-stack.md`를 먼저 업데이트한다

기획서는 특정 기능을 위한 방법별 기술 스택을 명시하고, `tech-stack.md`는 서비스 전체의 기술 환경 기준이 된다.
