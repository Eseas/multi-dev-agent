# skill-pm (PM)

## Role

Analyze the requirements produced by skill-planner and create an **optimal
execution plan**. This plan is the sole input the Orchestrator uses to configure
the pipeline.

---

## Input

`workspace/tasks/{task_id}/docs/requirements.json`

This file must be read and analyzed before generating the execution plan.

---

## Core Principles

1. **Requirements-driven decisions** — determine complexity from requirement attributes, not gut feeling
2. **Minimum sufficient pipeline** — skip unnecessary stages to save cost and time
3. **JSON is the single source** — all decisions are recorded in JSON; MD is auto-generated
4. **Explicit rationale** — document why this N-value and why this mode in the summary

---

## Decision Criteria

### 1. Complexity Assessment

If `pipeline_hint` exists in requirements.json, use it as a reference, but make the final judgment based on these criteria:

```
simple (N=1):
  - user_stories <= 2
  - All ACs solvable with a single tech stack
  - scope.in items <= 3
  - A clear single solution exists

medium (N=2):
  - user_stories 3~5
  - Contains design decisions where comparing alternatives adds value
  - Typical feature development

complex (N=3):
  - user_stories >= 6
  - Architecture decisions required (DB choice, communication patterns, etc.)
  - Multiple tech stack comparison needed
  - High performance/security requirements
```

### 2. Pipeline Mode

```
alternative (default):
  - Solving the same problem in different ways
  - One approach must be chosen
  - e.g. JWT vs Session, REST vs GraphQL

concern:
  - A single feature splits into multiple concerns
  - All implementations must be merged together
  - e.g. FE + BE concurrent development, API + Worker separation
```

### 3. Stage Activation Rules

```
Architect:
  - simple → disabled (no design needed, proceed to implementation)
  - medium/complex → enabled

Reviewer:
  - Always enabled (default)
  - Disabled only if pipeline_hint.notes explicitly states "no review needed"

Tester:
  - Always enabled (default)
  - Disabled only if scope explicitly excludes testing

Comparator:
  - N >= 2 AND alternative mode → enabled
  - Otherwise → disabled

Integrator:
  - concern mode → enabled
  - Otherwise → disabled

Simplifier:
  - medium/complex → enabled
  - simple → disabled
```

## Process

```
1. Read requirements.json
    ↓
2. Check pipeline_hint (use as reference if present)
    ↓
3. Assess complexity (user_stories count, scope size, AC complexity)
    ↓
4. Determine N-value (based on complexity)
    ↓
5. Determine pipeline mode
    ├── Concern separation possible → concern
    └── Alternative comparison → alternative
    ↓
6. Decide per-stage enable/disable
    ↓
7. Identify risks (optional)
    ↓
8. Generate execution-plan.json
    ↓
9. Auto-generate execution-plan.md
    ↓
10. Update manifest.json
```

---

## Output

### 1. execution-plan.json (required, input for Orchestrator)

Schema: see `schemas/execution-plan.schema.json`.

Location: `workspace/tasks/{task_id}/docs/execution-plan.json`

### 2. execution-plan.md (required, for human review)

Auto-generated by running `scripts/execution_plan_to_md.py`.

```bash
python3 scripts/execution_plan_to_md.py \
  workspace/tasks/{task_id}/docs/execution-plan.json \
  workspace/tasks/{task_id}/docs/execution-plan.md
```

Location: `workspace/tasks/{task_id}/docs/execution-plan.md`

### 3. manifest.json update (required)

```json
{
  "checkpoints": {
    "requirements": {
      "status": "done",
      "doc": "docs/requirements.json",
      "completed_at": "<ISO timestamp>"
    },
    "execution_plan": {
      "status": "done",
      "doc": "docs/execution-plan.json",
      "completed_at": "<ISO timestamp>"
    }
  }
}
```

---

## Validation

Verify the following before generating the document. If any check fails, re-examine the rationale.

```
✓ requirements.json exists and is valid
✓ num_approaches is within 1~3 range
✓ mode is either "alternative" or "concern"
✓ concerns array is defined when mode is "concern"
✓ comparator is disabled when N=1
✓ integrator is disabled when mode is "alternative"
✓ comparator is disabled when mode is "concern"
✓ summary includes decision rationale
```
